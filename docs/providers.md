# Providers

Status: Research

## Provider Classes

- `pi.dev` is the agent substrate and session runtime foundation
- `exe.dev` is the only supported deployment platform in v1
- Providers in Skarbot should be treated as four distinct classes:
  - model providers
  - channel providers
  - platform providers
  - service integrations

## V1 Model Provider Contract

- V1 should support exactly one model provider:
  - `openai-codex`
- V1 should use OpenAI Codex subscription auth rather than API-key auth
- The deployment default model should be `openai-codex/gpt-5.4`
- V1 should not define model fallbacks yet
- `exe.dev`'s LLM Gateway is intentionally out of scope for v1
- Adding Anthropic, Gemini, or other providers later should reuse the same model-config shape rather than introducing a second auth or routing system

## V1 Model Auth Ownership

- Model auth is the one planned exception to the general `exe.dev` proxy or integration rule in v1
- The Skarbot admin should log into the VM once to establish deployment-wide OpenAI Codex subscription auth
- That Codex auth should be shared by all Skarbot sessions in v1 rather than owned per user
- The canonical deployment-wide model-auth file should live at `~/.skarbot/auth.json`
- That model-auth file should be host-managed deployment state rather than repo config or portable Skarbot runtime state
- `~/skarbot/state` should remain portable and should not contain deployment model secrets
- Skarbot should not store raw model credentials in the repo
- Skarbot should not write raw model credentials into generated `TOML`
- Non-model service integrations should still prefer `exe.dev` proxy or integration-backed secrets in v1

## Model Selection and Persistence

- New threads should start from the deployment default model
- A thread should be allowed to switch models during its lifetime
- A thread's current model should be remembered per thread rather than as one mutable global user preference
- Model changes should be persisted in the thread session history, following the `pi-coding-agent` model-change pattern
- Restoring a thread should restore its last selected model when that model remains available
- V1 should not let one thread's model switch silently change the default model for every other thread
- The deployment default model should remain a protected runtime setting

## Model Switching Surfaces

- Terminal and web UI should support model switching in v1
- Direct-thread and task-thread sessions should both honor per-thread model selection
- The standard grouped model-control tool should be:
  - `model()`
  - `model(list)`
  - `model(set, provider, model)`
- `model()` should surface the current thread model plus thread usage and cost
- `model(list)` should show currently configured available models
- `model(set, provider, model)` should switch only the current thread and should persist that change in thread session history
- SMS and inbound email do not need their own dedicated model-picker UX in v1
- If a thread is resumed from a non-UI surface, Skarbot should continue using that thread's persisted current model

## Usage and Cost

- V1 should rely on `pi` session usage metadata as the canonical first-pass usage source
- Per-thread token and cost usage should be available from session history
- The current model should be visible in the thread or session UI
- Thread or session summaries should be able to show tokens and cost for that thread
- Provider-wide subscription usage or quota dashboards are not required for v1
- Richer provider-usage or billing snapshots can be added later as a separate observability layer

## Channel Providers

- ClickSend is the planned SMS provider for v1
- Inbound email should use the inbox already provided by the `exe.dev` VM
- Admin notifications may use outbound email sent from the `exe.dev` VM
- General outbound member email is out of scope for v1
- Web identity should use the authenticated email provided by the `exe.dev` proxy

## Platform and Service Integrations

- GitHub access should prefer `exe.dev` integrations over long-lived tokens on the box
- Service integrations such as ClickSend should prefer `exe.dev` proxy or integration endpoints so the app never stores raw API secrets when avoidable
- No additional day-one integrations are required beyond OpenAI Codex subscription auth, ClickSend, the `exe.dev` inbox, and `exe.dev` GitHub access

## Provider Rules

- No secrets should be stored in the repo
- No secrets should be written into generated `TOML`
- No local secrets file should be generated by setup for ordinary service integrations
- Deployment-wide model auth is the temporary v1 exception and should be established interactively on the VM by the admin into `~/.skarbot/auth.json`
- `exe.dev`'s authenticated email should be the default identity source for web membership and session checks in v1
